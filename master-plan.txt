# Clinical Note Hallucination Detector — Implementation Plan

## Context
Goal: Build an MVP web app that detects hallucinations in AI-generated clinical notes by verifying each claim against the original doctor-patient conversation transcript. This is a portfolio project targeting Abridge (AI medical scribe company, $5.3B valuation). The app demonstrates deep understanding of their core challenge: ensuring AI-generated medical notes don't contain fabricated information.

Tech stack: Next.js (App Router), TypeScript, React, Tailwind CSS, OpenAI API (GPT-4o).
Scope: Full Demo — clean UI, sample demo data, confidence scores, exportable report, landing page. No auth, no database.

---

## Architecture

Two-phase analysis pipeline:

  User pastes transcript + note
    -> [1] POST /api/extract-claims — breaks note into atomic claims
    -> [2] POST /api/verify-claims — checks each claim against transcript
    -> Color-coded results with evidence links

Why two calls, not one: extraction (decomposition) and verification (comparison) are different cognitive tasks. Separating them improves accuracy on both and gives the user progress feedback between steps.

State machine: idle -> extracting -> extracted -> verifying -> complete

## Build Phases

### Phase 1: Scaffold
- create-next-app with TypeScript + Tailwind + App Router
- Install openai package
- Create .env.local with OPENAI_API_KEY
- Set up folder structure

### Phase 2: Core Logic (most important — get AI working before UI)
Files:
- src/lib/types.ts — all TypeScript interfaces (Claim, VerifiedClaim, Verdict, Evidence, AnalysisResult)
- src/lib/openai.ts — OpenAI client singleton
- src/lib/prompts.ts — claim extraction prompt + grounding verification prompt + JSON schemas
- src/lib/claim-extractor.ts — calls OpenAI to break note into atomic claims
- src/lib/claim-verifier.ts — calls OpenAI to verify each claim against transcript
- src/app/api/extract-claims/route.ts — POST endpoint
- src/app/api/verify-claims/route.ts — POST endpoint

Prompt strategy (the core IP):
- Extraction prompt: decomposes note into atomic claims, each categorized (patient_statement, vital_sign, medication, diagnosis, etc.) and tagged with SOAP section
- Verification prompt: checks each claim with 8 explicit edge-case rules:
  1. Medical terminology translation (lay -> clinical) = grounded
  2. Clinical inferences only grounded if doctor said them
  3. Physical exam findings must appear in transcript
  4. Plan specifics (drug names, doses) must match exactly
  5. Vital signs must match exactly
  6. Negations only grounded if explicitly asked/denied
  7. Medication names/doses must match transcript
  8. Temporal claims (durations/dates) must match
- Confidence scores: 0.0-1.0 with calibrated rubric
- Uses OpenAI structured outputs (response_format: json_schema) for guaranteed valid JSON

### Phase 3: Demo Data
- src/data/demo.ts — realistic doctor-patient conversation + SOAP note with ~10 grounded, ~10 hallucinated, ~5 partial claims
- Hallucinations are subtle but detectable: wrong pain score (7->8), fabricated drug names ("blood pressure pill"->"lisinopril 10mg"), wrong follow-up timing (3 weeks->2 weeks), reversed diagnosis priority, added findings never discussed

### Phase 4: UI Foundation
- src/components/ui/ — Button, Card, Textarea, ProgressBar, Badge (Tailwind-only, no component library)
- src/app/layout.tsx — Inter font, header bar
- src/app/analyze/page.tsx — two-column layout (inputs left, results right)
- TranscriptInput, NoteInput, AnalysisControls components

### Phase 5: Analysis Flow
- src/hooks/useAnalysis.ts — state machine hook managing extraction -> verification flow
- SummaryStats — grounded/hallucinated/partial counts + accuracy gauge
- ClaimCard — individual claim with verdict badge, confidence bar, evidence quotes, expandable reasoning
- ResultsDashboard — filterable claim list
- TranscriptHighlight — transcript view with highlighted evidence lines, scroll-to on claim click
- Wire "Try Demo" button to preload demo data and auto-analyze

### Phase 6: Export + Polish
- src/lib/report-generator.ts — pure function generating markdown report (no LLM call)
- ExportButton — client-side download via Blob URL
- Loading animations (skeleton cards during extraction, spinner during verification)
- Error handling UI
- Responsive layout

### Phase 7: Landing Page
- Hero: "Every claim. Verified." + subheadline + CTA buttons
- HowItWorks: 3-step visual (Paste -> Analyze -> Verify)
- Features: claim-level granularity, confidence scoring, evidence linking, exportable reports
- CallToAction: "Built for clinical AI teams who refuse to ship hallucinations."

### Phase 8: Final Testing
- End-to-end demo flow
- Prompt tuning if verdicts are wrong
- Mobile responsive check
- Export validation
